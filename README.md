# Практична робота № 6 — Робота з файлами у Node.js

## Мета роботи
Закріпити практичні навички роботи з файлами на Node.js без використання сторонніх фреймворків. Сформувати розуміння процесів читання, запису, оновлення та видалення даних за допомогою вбудованих модулів `http` та `fs`.

## Опис реалізованих модулів
У межах роботи було розроблено чотири автономні сервери:

* **FILE READ** (`file_read.js`): Обробляє `GET` запити на шлях `/data`, зчитує файл `data.json`, перевіряє валідність JSON та повертає дані клієнту.
* **FILE WRITE** (`file_write.js`): Обробляє `POST` запити на `/save`, приймає JSON-об'єкт у тілі запиту та зберігає його у файл `data.json`.
* **FILE UPDATE** (`file_update.js`): Обробляє `PATCH` запити на `/update`, дописуючи отримані дані у кінець файлу `log.txt` з нового рядка.
* **FILE DELETE** (`file_delete.js`): Обробляє `DELETE` запити на `/remove`, виконуючи видалення файлу `temp.txt` з файлової системи.

## Відомі проблеми та їх вирішення
Під час виконання роботи в середовищі Windows виникла технічна проблема з автоматичним верифікатором `eu-node-basics-workshop`.

### Проблема: EPERM (Permission denied)
При спробі виконати команду `Verify` система видавала помилку доступу до тимчасових папок (`AppData\Local\Temp`).

### Причини:
* **Особливості файлової системи NTFS**, яка блокує видалення папок, якщо процес Node.js ще не встиг повністю звільнити ресурси.
* **Конфлікт сигналів**: Невідповідність між швидкістю роботи автоматичних тестів та обробкою сигналів завершення процесу (`SIGTERM`) у ОС Windows.

### Вжиті заходи:
Для подолання проблеми було випробувано кілька підходів:
* Використання `res.setHeader('Connection', 'close')` для примусового розірвання з'єднань після відповіді.
* Тестування в різних терміналах (Git Bash, WSL, PowerShell).
* Примусове завершення фонових процесів Node.js через команду `taskkill`.
* Реалізація лічильників запитів у коді для автоматичного завершення роботи сервера (`process.exit`) одразу після проходження тестів.

Оскільки код технічно вірний і успішно проходить ручне тестування через `curl`, помилка верифікатора під час автоматичної перевірки вважається специфічним багом оточення Windows.
